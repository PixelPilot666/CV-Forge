# 更新素材库（增量维护 master-profile.yaml）

素材库是长期资产，会随你的新经历不断增长。本文件定义如何**安全地增量更新** `~/.cv-forge/master-profile.yaml`，覆盖四类场景：新增、合并、修改、删除/归档。

> 触发：用户说「我新做了个项目」「换实习了，更新经历」「这个数字重测了，改一下」「把某段隐藏掉」等，或丢来一份新文档要求并入素材库。

## 通用原则（每次更新都遵守）

1. **合并，不覆盖**：在现有 YAML 基础上增改，绝不重写整个文件丢掉历史条目。
2. **稳定 id 不变**：已有条目/bullet 的 `id` 一律保持原样——简历裁剪、`tailor.json`、`check_fidelity` 都依赖它。新增条目才分配新 id。
3. **改动前先读、后校验**：编辑前先读当前 profile；写完跑 `python3 "$SKILL/scripts/validate_profile.py" ~/.cv-forge/master-profile.yaml`，确认 id 唯一、必填字段齐全。
4. **真实性边界照旧**：只录入用户确认的真实信息，缺数字就追问（见 interview.md），绝不编造。
5. **写前确认**：把要新增/修改/删除的内容摘要给用户，确认后再落盘。

## 场景一：新增经历 / 项目 / 论文

1. 判断归属 section：实习/工作→`experience`，项目→`projects`，论文→`research`，教育→`education`。
2. **分配不冲突的稳定 id**：
   - 条目 id 用语义化前缀，如 `expe-<公司缩写>`、`proj-<项目缩写>`、`pub-<关键词>`；与现有 id 比对确保唯一。
   - bullet id 在条目 id 后加 `-b1/-b2/...`。
3. 按 `references/profile-schema.md` 填字段（org/title/date 或 name/date、tech、tags、bullets，能量化的填 `metrics`）。
4. **位置**：插到该 section 内合适的时间位置（通常最新的放前面，见 resume-craft D2）。
5. validate → 给用户看摘要 → 写回。

## 场景二：从新文档抽取并合并

用户丢来新项目文档/源码/旧简历片段时：
1. 用与建库相同的方式抽取（`extract_profile.py` 适用于 LaTeX 简历；其他文档由 agent 阅读后结构化）。
2. **合并而非覆盖**：抽取结果作为「新增条目」并入现有库，按场景一分配新 id；不要用抽取结果替换整个文件。
3. 抽取常缺量化数字——按 interview.md 追问补全。
4. validate → 确认 → 写回。

## 场景三：修改现有条目

1. 先定位 id（如「把 expe-vast 的 caption 延迟数字改一下」）。
2. 只改该条目/bullet 的对应字段（text / metrics / tech / date 等）；**id 保持不变**。
3. 若改的是 §A1 类客观实体（公司名、岗位名、时间、论文名）——这通常意味着之前录错了，确认后可改，但要提醒用户这会影响已生成简历的 `check_fidelity` 基准。
4. validate → 确认 → 写回。

## 场景四：删除 / 归档旧条目

- **优先归档而非物理删除**，防止误删后无法恢复：
  - 临时不想展示某条 → 加进 `preferences.exclude_ids`（裁剪时跳过，但素材仍在）。
  - 确实过时要移除 → 跟用户确认后再从 YAML 删除该条目。
- 删除前检查：是否有 `skills.evidence_refs` 或别的引用指向它，避免留下悬空引用（validate 会提示）。

## 更新后

- 跑 validate 确认无误。
- 告诉用户更新了什么（新增/改了哪几条、新 id 是什么）。
- 提醒：若已有针对某 JD 的简历，可基于更新后的库重新生成。

## 与生成流程的关系

更新素材库是**独立动作**，可不接 JD 单独做（纯维护），也可发生在生成流程的第 2/3.5 步（加载校验、追问写回）。无论何时写回，都走本文件的原则。
