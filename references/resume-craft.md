# 简历写作准则（专家知识库）

裁剪与改写时**必须**加载本文件并遵守。面向计算机领域、**主攻校招（应届 + 实习）**，全岗位通用。

---

## A. 硬规则（违反即不合格，优先级最高）

### A1. 关键信息原样保留（verbatim，绝不改写）
以下字段必须与 master profile **一字不差**，不得改写、美化、归类、翻译：
- 个人信息：姓名、电话、邮箱
- 工作/实习经历：**公司名、岗位名（title）、起止时间**
- 科研经历：**论文名、发表处、时间、作者位次**
- 项目经历：**项目名、时间、链接**

> 反例（禁止）：把「Agent 工程师」改写成「算法工程师（推荐方向）」。岗位名是事实，只能原样。
> 可以做的：在 **bullet 描述里**用 JD 词汇组织内容（见 B），但 heading 中的公司/岗位/时间不动。

### A2. 整页纪律
- 目标：**尽量正好 N 整页**（校招应届/实习**最好 1 页**），不能出现明显半页空白。
- 就近取整：
  - 偏多（如 1.2~1.4 页）→ **压缩**：删次要 bullet、合并、精简措辞，压回 1 页。
  - 偏少（如 0.6~0.8 页）→ **补满**：补充内容到铺满，**但最多只能加一种**：`个人特点` / `技能专长` / `兴趣爱好`（三选一，优先级：技能专长 > 个人特点 > 兴趣爱好）。
  - 差距很大（如 1.7 页）→ 优先压缩到 1 页；确实压不动再考虑铺到 2 整页。

### A3. 排版饱满
- 每条 bullet 尽量**铺满整行**，句尾不留大段空白；过短的点合并或补充细节，过长的点拆分或精简。
- 避免「孤行」（一条 bullet 只占半行尾巴）。

### A4. 语言简练专业 + 适度加粗
- 简练：去掉「负责了一个……的工作」这类废话，直接动词开头讲成果。
- 加粗（`\textbf{}`，写在 bullet_overrides 的纯文本里时**不要**手写 `\textbf`，改用约定标记见 §E）：只加粗**量化指标、关键成果、核心技术名**，**不整句加粗**，一条 bullet 加粗不超过 1~2 处。

## B. Bullet 写作公式

**强动词 + 做了什么 + 怎么做（技术/方法）+ 可量化的结果**

- 弱：参与推荐系统开发，负责部分模块。
- 强：**主导**召回-排序全链路优化，基于 BGE-M3 + RRF 实现多路混合召回，**P@1 提升 10pct**。

### 动词分级（开头动词决定第一印象）
- 强（主导性）：主导、设计、构建、提出、优化、重构、落地、搭建、实现
- 中（参与性）：开发、完成、负责、支持
- 弱（避免）：参与、帮助、协助、了解、学习、接触

### 成果导向 > 职责导向
- 职责：负责模型训练与评测。
- 成果：将 P@1 从 85% 提升至 95%，建立可量化离线评测体系。

### 项目经历用 STAR 原则
**项目段**的描述要体现 STAR，让读者在 2~3 条 bullet 内看清「背景—任务—行动—结果」：

- **S 情境 Situation**：在什么场景/为解决什么问题（一句带过，不展开）。
- **T 任务 Task**：你要达成的目标 / 承担的角色。
- **A 行动 Action**：用什么技术/方法做了什么（这是技术含金量所在，写最重）。
- **R 结果 Result**：可量化的成果 / 影响（指标、规模、效率）。

落地方式（不必逐字写 S/T/A/R，而是让信息齐全）：
- 首条 bullet 交代 **S+T**：「面向 X 场景，构建 Y 系统，解决 Z 问题」。
- 中间 bullet 写 **A**：分点讲关键技术决策与实现（每条仍是强动词开头）。
- 结尾 bullet 给 **R**：用真实数字收口；无数字则触发追问（见 interview.md）。

> 示例（多Agent选课推荐系统）：
> - **S+T**：面向高校教务选课场景，构建多 Agent 选课推荐系统，解决学生在海量课程中按兴趣/时间/校区个性化选课的难题。
> - **A**：基于 LangGraph 设计 Supervisor 多 Agent 编排，将推荐链路拆为画像→召回→重排→校验→解释五阶段；构建 MySQL + Milvus 课程 RAG 数据层，Redis 缓存候选集降低检索开销。
> - **R**：（待补真实数字：覆盖课程数 / 推荐准确率 / 检索耗时）。

## C. 量化纪律

- 每条 bullet **尽量带数字**：指标（P@1/准确率/GMV/延迟）、规模（数据量/QPS/用户数/候选数）、效率（耗时/提升幅度/成本下降）。
- 没有数字的 bullet 是**追问访谈的目标**（见 references/interview.md）——主动逼出数字。
- 数字必须真实，来自 profile 的 `metrics` 或经用户确认的事实；**绝不编造**（见 tailoring-rules.md）。
- 百分号等在纯文本里正常写（`96.7%`），由 fill_template 自动转义；不要手写 `\%`。

---

## D. 结构与顺序（校招）

### D1. 章节顺序（section 之间）
- **教育经历居首**（校招惯例，固定）。
- **实习经历与项目经历相邻摆放**（二者同属"工程实践"，放在一起，实习在前、项目紧随）。
- **科研经历的位置按与 JD 的相关性决定**：
  - 科研比项目更贴合 JD（如 JD 偏算法/研究/对口论文方向）→ 顺序为：**教育 → 科研 → 实习 → 项目**。
  - 否则（JD 偏工程/应用）→ 顺序为：**教育 → 实习 → 项目 → 科研**。
- 末尾可选补充：技能 / 个人特点 / 爱好（整页纪律 A2，最多补一种）。

### D2. 块内子内容排序（每个 section 内部）
1. **先按时间倒序**（最近的在最前）。
2. **同一时间或无明确时间时，再按与 JD 的相关性排序**（更相关的在前）。
- 实习/工作：按起止时间倒序；并列时最贴合 JD 的在前。
- 项目/科研：有时间的按时间倒序；无时间（如个人项目、在投论文）的，按 JD 相关性排序。
- 教育：按时间倒序（硕士在本科前）。

### D3. 其他
- **6 秒原则**：每个 section 内最相关、最亮眼的内容尽量靠前（在不违反 D2 时间倒序的前提下，用相关性决定并列项次序）。
- 与 JD 无关的内容狠心砍。
- **长度**：应届/实习 1 页为佳（见 A2）。

## E. 与 fill_template 的协作约定（重要，防止转义事故）

- `bullet_overrides` / bullet `text` 里写**纯文本**，`%`、`&`、`_` 等由脚本自动转义，**不要手写** `\%`、`\textbf{}`。
- **bullet 正文加粗**：用 markdown 风格的 `**要加粗的内容**`。脚本会先转义再把成对的 `**…**` 转成 `\textbf{…}`，安全无转义冲突。
  - 例：`"P@1 从 **85% 提升至 95%**，建立离线评测体系"` → 渲染为粗体的指标。
  - 克制：一条 bullet 的 `**…**` 不超过 1~2 处，只圈量化指标/关键成果/核心技术名，**不整句加粗**。
- `heading` / `date` / `tech_line` 是 raw 字段，可写排版 LaTeX（`\textbf{}` 等），但**其中的公司名/岗位名/时间必须 verbatim**（A1）。

## E2. 专业语言规范（让简历读起来像资深工程师写的）

简历语言要**精确、克制、术语规范**，避免口语化、空泛、营销腔。逐条对照：

### 用精确的技术动词，而非泛化动词
| 泛化（弱） | 专业（强） |
|---|---|
| 做了一个推荐系统 | 设计并实现召回-排序推荐链路 |
| 用了 RAG 技术 | 基于 RAG 构建检索增强生成流程 |
| 提升了效果 | 将 P@1 提升 10pct / 召回准确率提升至 96.7% |
| 优化了性能 | 将 P99 延迟从 800ms 降至 120ms |
| 处理数据 | 构建 ETL 链路 / 清洗并标注 X 万条样本 |

### 去掉空泛与口语
- 删除：「负责了……的相关工作」「做了一些……」「主要是……」「等等」「非常/很/比较」。
- 删除主观自夸形容词：「优秀的」「强大的」「高效的」——用**数字**证明，而不是用形容词宣称。
- 反例：「搭建了一个非常高效的推荐系统」→ 正例：「搭建推荐系统，单次组品 42s 内完成 105+ 候选筛选」。

### 去 AI 味（让文字像人写的）
- **避免「概括 + 冒号 + 展开」的套路句式**：这是最典型的 AI 腔。
  - 反例：「主导从 0 到 1 的架构设计与全部开发：以 LangGraph 编排……」
  - 正例：「独立完成系统的架构设计与开发，用 LangGraph 编排多 Agent 推荐链路……」（直接陈述，不用冒号转折）
- **描述信息里不要用括号补充解释**：括号里的内容要么并进正文，要么删掉。
  - 反例：「原子化 upsert（先校验写入再清理旧块）」「双轨混合检索（dense + sparse）」
  - 正例：「先校验写入成功再清理旧块的原子化 upsert」「dense 与 sparse 双轨混合检索」
  - 例外：技术名缩写首次出现的全称、岗位/学校的客观限定（如学校层次）等**事实性**括注可保留，但**解释性**括注一律去掉。
- **不要用破折号（—— / —）**做解释或转折，改成正常句子或逗号分句。
- **避免过于绝对的表达**：「绝不丢数据」「100% 准确」「完全解决」「永不」这类词，改成有分寸的客观表述（如「以数据不丢失为核心约束」「显著降低」）。
- 避免空洞的拔高词：「赋能」「闭环」「抓手」「全链路打通」「端到端能力」堆砌；能用具体动作说清就别用大词。
- 避免排比凑数与对仗腔（「不仅……而且……」「既……又……」），简历是陈述不是演讲。
- 一句话只讲一件事，主谓宾清楚；读起来顺口、像同行口头讲项目，而不是营销文案。

### 公司/产品介绍（不知名公司）
- 若公司不够知名，可在公司名后或简介里用**极短一句**点出它的代表产品或行业地位，帮 HR 建立认知。
- 信息来源：联网检索或用户已有信息；只写**可核实的客观事实**，不吹捧。
- 形式可以是公司名旁注代表产品，如 `XX 科技（某旗舰产品）`；或简介首句一句话带过（如「国内领先的 XX 平台」）。知名大厂（字节、腾讯等）无需介绍。
- **主语必须清楚，公司介绍不能和"我做了什么"混在一句里**：公司介绍的主语是公司/产品，个人职责的主语是本人（简历里不出现"我"，靠动词开头表达）。两者**分成独立的句子**，否则会读成"公司在负责"。
  - 反例：「XX 是国内领先的某平台。负责检索内核……」（第二句主语断裂，像公司在负责）
  - 正例：把公司介绍放进**标题旁注**（`XX 科技（某旗舰产品）`），简介只写个人职责：「负责检索与文档摄取两大内核……」（主语自然是本人）。
  - 或正例：简介首句以本人动作开头，公司背景作前置状语：「在国内领先的某平台 XX，负责检索内核……」。

### 动词不可夸大原创性（重要）
应用业界已有方法/范式/框架时，**不要**用「提出 / 首创 / 发明 / 创新性地设计了」这类词——会让人误以为该方法是你原创发明的，面试一问即穿。
- 用「**应用 / 采用 / 基于…实现 / 落地 / 引入 / 复现**」描述对已有技术的工程应用。
- 「**提出 / 设计**」只用于**确实由你原创**的方法（如你论文里真正提出的框架）。
- 反例：「提出『先检索后生成』范式」（RAG 是业界已有范式，非你提出）→ 正例：「采用『先检索后生成』范式，落地生成式组品流程」。
- 反例：「提出全图前向、关键区域反向的训练机制」（这是论文方法，措辞要贴合你的真实贡献）→ 若论文确为你原创可用「提出」，否则用「实现/复现」。

### 术语准确、中英规范
- 技术名用业界标准写法与大小写：`FastAPI`/`LangGraph`/`Milvus`/`Kubernetes`/`PyTorch`，不要 `fastapi`/`milvus`。
- 中英文之间留空格（`基于 RAG 构建`），数字与单位规范（`42s`、`96.7%`、`105+`）。
- 缩写首次出现可点明全称（如「检索增强生成（RAG）」），后续用缩写。
- 同一份简历术语统一：要么全用「召回」，不要一处「召回」一处「检索」混用指同一概念。

### 句式：动词开头、客观、无人称
- 不写「我负责……」「我们团队……」——简历默认主语是你，直接动词开头。
- 时态/句式统一：每条都是「动词 + 对象 + 方法 + 结果」的客观陈述，不用感叹/反问。
- 一条 bullet 一个核心意思；过长（超过 2 行半）应拆分或精简。

### 自检
- [ ] 有没有泛化动词（做、用、搞、负责）可换成精确动词？
- [ ] 有没有靠形容词自夸、却没有数字支撑的句子？
- [ ] 技术名大小写/中英空格是否规范统一？
- [ ] 有没有「我/我们」「等等」「相关工作」这类口语？
- [ ] 有没有「概括+冒号+展开」的 AI 腔句式、或「赋能/闭环/抓手」等空洞大词？
- [ ] 有没有解释性括号、破折号、或「绝不/100%/永不」这类过于绝对的表达？
- [ ] 不知名公司是否用极短一句点明代表产品/地位（仅客观事实）？

## E3. 技术栈规范（什么能进、什么不能进）

**定义**：技术栈是候选人**有实际使用经验、能独立完成相关开发任务**的技术集合——属于**技术能力**，不是项目成果。用于快速判断技术方向与能力边界。

### 可以进技术栈（按需分类，命中 JD 的类别在前）
- 编程语言：Python、Java、C++、Go、TypeScript、SQL…
- 前端：Vue3、React、Next.js、Element Plus、ECharts…
- 后端：Spring Boot、FastAPI、Flask、Gin、Node.js…
- 数据库 / 中间件：MySQL、PostgreSQL、Redis、MongoDB、Elasticsearch、Kafka…
- AI / 大模型：PyTorch、Transformers、LangChain、LlamaIndex、vLLM、SGLang、RAG、Agent、MCP…
- 数据工程：Pandas、NumPy、Spark、Airflow…
- DevOps / 云原生：Linux、Docker、Kubernetes、Git、Nginx…
- ML / CV：Scikit-Learn、XGBoost、OpenCV、MMCV、YOLO、SAM…
- 向量库 / 检索框架 / 可观测等具体中间件：Qdrant、Milvus、FAISS、Haystack、LiteLLM…

### **不能**进技术栈（Agent 必须过滤掉）
- ❌ **项目名称**：智慧旅游平台、电商推荐系统、多Agent选课推荐系统
- ❌ **产品名称**：ChatGPT、Claude、Cursor（除非确为你日常使用的工具栈，且单独标注）
- ❌ **荣誉奖项**：国家奖学金、ACM 银奖
- ❌ **论文 / 专利**：CVPR 论文、发明专利
- ❌ **软技能**：沟通能力、团队协作、学习能力
- ❌ **工作内容 / 能力描述**：召回与排序、系统设计（这些写进 bullet，不是技术栈）
- ❌ **语言内置库 / 标准库模块**：asyncio、threading、json、os（属于「会用该语言」的范畴，不与框架/中间件并列；写「Python」即可，必要时在 bullet 里点明「基于 asyncio 并发」）

### 算法 / 范式的归属（重要）
像 BM25、RRF、RAG、Multi-Agent、LoRA 这类**算法/范式/方法**：
- 若是该岗位的**核心技术方向标签**（如 AI 岗的 RAG、Agent、MCP），可少量进技术栈。
- 但**具体算法细节**（如 RRF、DBSF、small-to-big）更适合写进 bullet 体现「实际用过、懂取舍」，而非堆在技术栈行——技术栈行只放最能代表能力边界的项。

### 长度
- **技术栈只保留约一行**（铺满一行、不折行到孤行；见 A3）。挑最能命中 JD、最代表能力的项，多余的删。

### 只写本人主要参与的部分（重要）
- 技术栈与 bullet **只能写你本人主要参与/负责的部分**。团队项目里别人做的模块、你只旁观或了解的技术，**不写进你的技术栈，也不据为成果**。
- 判断标准：你能在面试里讲清「为什么这么做、怎么做的、遇到什么问题怎么解决」——能讲清才算你主要参与。
- 这与真实性硬边界一致：技术栈是"能独立完成相关开发任务"的能力，不是"项目里出现过的所有技术"。

### 项目简介（放在标题正下方，作为该段开始）
- **"项目"是广义的**：不只"项目经历"段。**实习/工作经历里负责的项目、实践项目、科研项目，乃至任何一段独立的工作内容，都算项目，都要有简介。**
- 位置：紧接该条目 heading 之后、要点列表之前，写**一段连贯的专业文字**（非要点、非罗列）。
- 内容：1~2 句讲清这个项目是什么 + **你本人的核心亮点与贡献**（STAR 的 S+T，点出你主导/负责的关键部分）。
- 简介聚焦"你做了什么、价值在哪"，技术细节留给下面的要点展开。

## F. 岗位差异化（全岗位通用，按 JD 类型调整强调点）

| 岗位类型 | 重点强调 | 弱化 |
|---|---|---|
| 算法 / AI / 推荐 | 指标提升、建模方法、论文、数据规模、评测闭环 | 纯 CRUD 业务 |
| 后端 / 基础架构 | 高并发、QPS、延迟、架构设计、稳定性、规模 | 前端样式 |
| 前端 / 客户端 | 性能优化、组件化、工程化、用户体验指标 | 后端细节 |
| 数据 / 数仓 | 数据规模、链路、口径、看板、降本 | 与数据无关项 |
| 产品 / 运营 | 业务结果、增长指标、用户量、转化 | 底层技术细节 |

> 差异化只改变**强调与排序**，不改变事实，也不改写 A1 中的原样字段。
